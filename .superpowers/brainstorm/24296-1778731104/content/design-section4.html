Brio Theme — Data Flow

Section 4/5: How data flows between dashboard, backend, and storefront

Flow Diagram

Dashboard → Backend → Storefront
1. Merchant edits theme in Dashboard
PUT /api/v1/merchant/theme
Body: { theme_name, hero_headline, hero_image_url, story_text, featured_product_ids, contact_phone, ... }
↓
2. Backend validates + saves to DB
store_theme_settings table updated
Redis cache invalidated: del theme:{storeId}
↓
3. Customer visits storefront
GET /store/{slug}/brio or GET /api/v1/public/stores/{slug}/theme
Server checks Redis cache first, falls back to DB
↓
4. Storefront renders with theme data
Brio theme components use theme data + products API to render page

API Endpoints Summary

EndpointScopePurpose
GET /api/v1/public/stores/:slug/themepublicRead theme settings (cached)
PUT /api/v1/merchant/thememerchantUpdate theme settings

Caching Strategy

  • Redis key: theme:{storeId}
  • TTL: 5 minutes (theme settings change rarely)
  • Invalidation: On merchant PUT, delete key immediately
  • Products: Featured products fetched separately via existing /products API (already cached)

Route Resolution

# Customer visits
/store/{slug}           → checks theme settings, redirects to /store/{slug}/brio if theme=brio
/store/{slug}/brio      → renders Brio theme
/store/{slug}/brio/menu → renders Brio category grid

# API calls from storefront
/api/v1/public/stores/{slug}/theme    → theme settings
/api/v1/public/stores/{slug}/products → products
/api/v1/public/stores/{slug}/categories → categories
