## Fixed During This Audit

### [FIXED] Staff Invitation JWT Missing `type` + `jti`
- **File:** `apps/backend/src/modules/staff/staff.route.merchant.ts:180`
- **Issue:** Staff invitation accept route generated JWT without `type: 'access'` and `jti`. Merchant scope hook checks `decoded.type !== 'access'`, causing all authenticated requests from newly-accepted staff to fail with 401.
- **Fix:** Added proper access token + refresh token with `type`, `jti`, stored refresh in Redis, set both cookies using standard `cookieOptions`.
- **Verification:** `pnpm typecheck` passed (0 errors).

### [FIXED] Dashboard Service External IP Routing
- **File:** `docker-compose.prod.yml`
- **Issue:** Dashboard service in Docker used external IP via `.env.production`, causing requests to route through Caddy instead of internal Docker network.
- **Fix:** Added `API_BASE_URL: http://spaceship_backend:3000` to dashboard service environment.

## Auth & Security Audit

### JWT & Authentication — PASS
- All 4 scopes (public, merchant, customer, superAdmin) have proper JWT verification
- `type: 'access'` / `type: 'refresh'` enforced in all scope hooks
- Token rotation with Redis-backed refresh tokens
- `jti` (JWT ID) used for revocation tracking
- Access token TTL: 15 minutes, Refresh token TTL: 7 days
- Cookies: httpOnly, secure in production, sameSite strict, signed
- No JWT returned in response body

### CSRF Protection — PASS
- CSRF token cookie set on safe methods (GET/HEAD/OPTIONS)
- CSRF validated on mutating methods (POST/PATCH/PUT/DELETE)
- Cookie readable by JS (httpOnly: false), strict sameSite

### Rate Limiting — PASS
- Login routes: 5/minute
- Register routes: 3/minute
- Staff invite: 5/minute
- Checkout: 5/minute

### API Key Auth — PASS
- SHA-256 hashing of keys
- Only keyPrefix exposed in list responses
- rawKey returned ONLY on creation
- Scope validation enforced
- Expiration check
- lastUsedAt tracking

### Password Security — PASS
- bcrypt with salt rounds 12
- Change password requires current password verification
- Password regex: uppercase, lowercase, number, special character

### Store Status & Plan Checks — PASS
- All authenticated scopes check store status (`active` only)
- Plan expiration checked in merchant + customer scopes
- Plan expiry warning headers (`x-plan-expires-soon`) sent when within 7 days

## Data Layer & API Audit

### Tenant Isolation — PASS
- Every DB query filtered by `storeId`
- `storeId` from JWT ONLY — never from body/query params (verified via grep: 0 matches)
- Payment webhook handlers verify `payment.storeId !== storeId`

### Inventory & Race Conditions — PASS
- `decrementInventory`: Atomic SQL decrement with `currentQuantity >= quantity` guard
- `decrementVariantOptionStock`: Same pattern for variant stock
- `restoreInventory`: Atomic increment on order cancellation
- Payment webhooks decrement inventory inside transaction with payment status update

### Order Number Collision — PASS
- `generateOrderNumber()` uses `Date.now().toString(36) + crypto.randomBytes(4).toString('hex')`
- `MAX_RETRIES = 3` with unique constraint catch on `23505`

### Pricing & Checkout — PASS
- Server-side price computation via `pricingService.computeOrderPricing()`
- Decimal math using integer cents (`toCents`, `fromCents`)
- Client never sends price/total — all computed from DB
- Inventory check before adding to cart
- Cart ownership verification (customerId vs cart.customerId)

### Coupon Logic — PASS
- Atomic usage increment with `usageLimit` guard inside SQL
- Per-customer usage tracking with limit check
- Date validation (startsAt, expiresAt)
- Minimum order amount check using decimal cents
- Product applicability check for product-specific coupons

### Payment — PASS
- AES-256-GCM encryption for provider configs
- No legacy plaintext fallback (throws error if detected)
- Webhook signature verification with `crypto.timingSafeEqual`
- Idempotency keys on all provider API calls
- COD marked as completed immediately in transaction
- Duplicate payment intent prevention (checks existing processing/completed)

### Error Handling — PASS
- Centralized error handler in `index.ts`
- Sentry integration for all errors
- Zod validation errors mapped to 400
- Custom error codes mapped to correct HTTP status codes
- Production: 500 errors hide message (`'An error occurred'`)
- Pino logger redacts: password, token, cardNumber, cvv, secret, apiKey, webhook_secret
