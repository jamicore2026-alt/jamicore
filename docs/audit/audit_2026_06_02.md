## User-Reported Bugs (Phase 1 — FIXED ✅)

These were the entry point for the audit. Both fixed, tested, and committed before the audit fan-out.

### BUG-001 — Settings page "Failed to load admin profile" ✅ FIXED

- **Symptom:** Super admin `/admin/settings` page showed "Failed to load admin profile".
- **Root cause:** `GET /api/v1/admin/auth/me` did not return `lastLoginAt`. The page read it, so the response shape mismatch surfaced as an error.
- **Fix:** `apps/backend/src/modules/auth/auth.route.superAdmin.ts:214-227` — added `lastLoginAt` to the response, mirroring the merchant `/me` shape.
- **Test fix:** `apps/backend/src/modules/auth/auth.route.superAdmin.test.ts` — added `lastLoginAt` to the mock.
- **Commits:** `97f9e3d` (fix) + `f3f2e73` (test).
- **Verification:** `pnpm typecheck` (0 errors), `vitest run` passes.

### BUG-002 — User dropdown overlap with "+ Add Category" ✅ FIXED

- **Symptom:** Merchant dashboard topbar user dropdown (Profile / Sign out) overlapped with page-level action buttons.
- **Root cause:** shadcn-svelte `DropdownMenuContent` ships with baked-in `z-50`. Other floating elements (FAB, page actions) also at z-50, leading to DOM-order wins that put the action button on top.
- **Fix:** `apps/dashboard/src/lib/components/layout/DashboardTopbar.svelte:89` — added explicit `z-[60]` to lift the dropdown above page actions.
- **Caveat surfaced by audit:** the ad-hoc fix only patches one call site. See **UI-001, UI-002, UI-003** below for the systematic fix (primitive-level z-index bump or design-token system).
- **Commit:** `ee8915c`.

## P1 — Should fix this sprint (13)

Ordered by dimension, then severity of impact. All have `file:line` evidence and a fix sketch in the corresponding JSON.

### Security (1)

#### SEC-001 — MFA code generated with `Math.random()` (NOT cryptographic)
- **File:** `apps/backend/src/modules/auth/auth.service.ts:490`
- **Impact:** Theoretical — 6-digit code space is 10⁶; rate limit 5/min/IP makes enumeration impractical. But the `Math.random()` call violates the "no insecure RNG" baseline a PCI-DSS audit would flag.
- **Fix:** replace with `crypto.randomInt(100000, 1000000)`. Optionally use `crypto.timingSafeEqual` for the compare.
- **Effort:** 5 min.

### Performance & DB (4)

#### PERF-001 — Abandoned-cart processor loads ALL products in the store per job
- **File:** `apps/backend/src/services/abandonedCartProcessor.service.ts:38-41`
- **Impact:** With 10k SKUs the job reads 10k rows per run. BullMQ runs it every 15 min.
- **Fix:** batch-load only the products referenced by the cart items: `SELECT id, titleEn FROM products WHERE id = ANY($1) AND storeId = $2`.
- **Effort:** 1 h.

#### PERF-002 — `mergeCartOnLogin` does N × `addItem` (N+1 across cart writes)
- **File:** `apps/backend/src/modules/cart/cart.service.ts:66-80`
- **Impact:** Each `addItem` fires 2-3 DB calls. For a 10-item cart at login that's 20-30 round-trips per login.
- **Fix:** batch-load products, single INSERT for cart items, single SQL aggregate for totals. Wrap in a transaction.
- **Effort:** 3 h.

#### PERF-003 — Public guest order creation does a per-item `productRepo.findById` (N+1)
- **File:** `apps/backend/src/modules/order/order.route.public.ts:54-79`
- **Impact:** `productRepo.findById` loads with deep relations (variants, modifierGroups). 5-item cart = 5 expensive calls. Fires on the hot path (checkout).
- **Fix:** `SELECT * FROM products WHERE id = ANY($1) AND storeId = $2` + Map lookup.
- **Effort:** 2 h.

#### PERF-004 — Product variant/option mutations don't invalidate the public product cache
- **File:** `apps/backend/src/modules/product/product.route.merchant.ts:135-243`
- **Impact:** Merchant edits a variant price or option name → storefront shows stale data for up to 10 min. The other product mutations (POST /, PATCH /:id, DELETE) DO invalidate correctly.
- **Fix:** every variant/option/modifier mutation must call `cacheService.deletePattern(\`products:public:${storeId}:*\`)`.
- **Effort:** 1 h (multiple call sites, mechanical change).

### Code Quality (2)

#### QUAL-001 — Error responses missing `code` field in super-admin order route
- **File:** `apps/backend/src/modules/order/order.route.superAdmin.ts:53`
- **Impact:** `GET /admin/orders/:id` returns `{ error: 'Order not found' }` without `code: ErrorCodes.ORDER_NOT_FOUND`. Clients can't match on a stable identifier. Other branches in the same module use ErrorCodes correctly — this is inconsistency, not systemic.
- **Fix:** import ErrorCodes and add `code: ErrorCodes.ORDER_NOT_FOUND`.
- **Effort:** 15 min.

#### QUAL-002 — Magic string `REVIEW_NOT_OWNED` not in `ErrorCodes`
- **File:** `apps/backend/src/modules/review/review.route.customer.ts:60,78`
- **Impact:** `code: 'REVIEW_NOT_OWNED'` is a string literal that doesn't exist in `errors/codes.ts`. The central error handler won't map it to a status, so the response uses an inline 403 by accident. Clients matching on code get a value not in the published contract.
- **Fix:** add `REVIEW_NOT_OWNED: 'REVIEW_NOT_OWNED'` to ErrorCodes, import and use the constant, add a 403 mapping in `index.ts` `codeToStatus`.
- **Effort:** 15 min.

### UI / UX (3)

#### UI-001 — AdminTopbar notification + user dropdowns lack z-[60] override
- **File:** `apps/dashboard/src/lib/components/layout/AdminTopbar.svelte:269,322`
- **Impact:** Same latent bug as BUG-002. The fix in BUG-002 only patched DashboardTopbar; AdminTopbar's two DropdownMenuContent instances still inherit `z-50` from the primitive.
- **Fix:** add `z-[60]` to both (or apply UI-003's primitive-level fix and remove all ad-hoc overrides).
- **Effort:** 5 min.

#### UI-002 — NotificationBell dropdown z-index not bumped
- **File:** `apps/dashboard/src/lib/components/notifications/NotificationBell.svelte:168`
- **Impact:** Same class of bug. Notification dropdown can be hidden behind z-50 dialog overlay if a dialog opens while it is open.
- **Fix:** add `z-[60]` to DropdownMenuContent at line 168.
- **Effort:** 5 min.

#### UI-003 — `dropdown-menu-content` primitive z-50 collides with dialog/drawer/cookie-banner
- **File:** `apps/dashboard/src/lib/components/ui/dropdown-menu/dropdown-menu-content.svelte:26`
- **Impact:** Primitive sets z-50 via `cn()`. Conflicting z-50 elements exist: `dialog/dialog-overlay`, `drawer/drawer-overlay`, `select-content`, `tooltip-content`, `popover-content`, `cookie-banner`. The BUG-002 fix is point-patching; this primitive fix prevents the same bug class in every future DropdownMenu consumer.
- **Fix (recommended):** bump primitive to `z-[60]` (or `z-[55]`) so it sits above dialogs by default. Then remove the ad-hoc `z-[60]` overrides from individual callers. Alternative: define a z-index token system (`--z-dropdown`, `--z-dialog`, `--z-toast`).
- **Effort:** 30 min including the cleanup of the point-patches.

### Cross-scope consistency (3)

#### CONS-001 — JWT payload / `/me` response shape diverges across 4 auth scopes
- **File:** `apps/backend/src/modules/auth/auth.route.merchant.ts:321-336, auth.route.customer.ts:327-333, auth.route.superAdmin.ts:214-227`
- **Impact:** Three different top-level keys (`{user, store}`, `{customer}`, `{admin}`), three different field sets, and the customer endpoint does not return `storeId` even though the token carries it. The superAdmin `/me` returns `name` and `lastLoginAt` that merchant `/me` does not — BUG-001 fix added `lastLoginAt` to superAdmin only.
- **Fix:** pick one canonical shape (recommend `{id, email, name?, role?, isActive?, lastLoginAt?, scope}` + nested `store` for merchant) and apply in all three `/me` handlers. Long-term: add `name` and `lastLoginAt` columns to the `users` table, update on merchant login.
- **Effort:** 4 h (incl. migration for `users.name` / `users.lastLoginAt`).

#### CONS-006 — Error response shape missing `code` or `message` field in merchant product/shipping routes
- **File:** `apps/backend/src/modules/product/product.route.merchant.ts:258,266,274, shipping.route.merchant.ts:37`
- **Impact:** CSV import 400s (No file uploaded, CSV empty, missing columns) lack `code`. ZONE_NOT_FOUND 404 lacks `message`. Convention is `{error, code, message}` everywhere else; these are outliers.
- **Fix:** add the missing field to each. Optionally add an ESLint rule that flags `reply.status(N).send({...})` calls missing any of `error/code/message`.
- **Effort:** 30 min.

#### CONS-007 — Plan expiry check is missing on public scope routes
- **File:** `apps/backend/src/scopes/public.ts:88-100, modules/auth/auth.route.merchant.ts:140-147, auth.route.customer.ts:64-71`
- **Impact:** Merchant and customer scope hooks reject with `PLAN_EXPIRED` when the store's plan is past expiry. Public routes (product, order, cart, etc.) run BEFORE any auth hook and never check plan expiry. A customer can browse products and create guest orders on an expired merchant plan. The 2026-05-21 audit "Plan expiration checked in merchant + customer scopes" does NOT cover the public storefront flow.
- **Fix:** add the same `store.status` + `planExpiresAt` check to the public scope hook. Optionally pass through with `x-plan-expires-soon` headers.
- **Effort:** 2 h.
